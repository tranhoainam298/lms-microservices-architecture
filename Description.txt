LMS MICROSERVICES ARCHITECTURE — CURRENT IMPLEMENTATION DESCRIPTION
==================================================================

Purpose
-------
This file describes the executable repository. The final Word architecture constraints,
current secure source code, API/event contracts, and verified runtime take precedence over
older examples. Conceptual endpoint names from the Word document are mapped to the actual
routes documented under shared/api-contracts and docs/architecture.

IMPLEMENTED NOW
===============

1. Core components
------------------
The LMS has exactly these deployable core components:

- API Gateway: Express.js, port 3000 inside Docker.
- User Service: Node.js/Express, port 5001.
- Course Service: Node.js/Express, port 5002.
- Exam & Quiz Service: ASP.NET Core / EF Core, port 5003.
- Payment Service: Node.js/Express, port 5004.
- Web Client: React/Vite production build served by Nginx.

There is no Analytics Service, Reporting Service, Notification Service, Enrollment Service,
or AI core service. Reporting responsibilities remain inside existing data owners. The AI
Chatbot System and Payment Gateway are external systems. RabbitMQ is infrastructure.

2. Database ownership
---------------------
Each business service uses one independent MySQL 8 database:

- User Service -> User DB: users, login_audit.
- Course Service -> Course DB: courses, lessons, enrollments, lesson_progress.
- Exam & Quiz Service -> Exam DB: quizzes, questions, quiz_results.
- Payment Service -> Payment DB: transactions.

No service connects directly to another service's database. There is no shared application,
Reporting, Enrollment, Learning Result, or Chatbot database. Cross-domain data is obtained
through authenticated HTTP APIs or RabbitMQ events.

3. Public request path
----------------------
The normal browser path is:

Browser -> http://localhost:8080 -> Nginx -> /api -> API Gateway -> owning service.

Nginx never routes /api directly to a business service. API Gateway mounts /auth, /users,
/courses, /exams, and /payments. The Web Client uses a relative /api base and never calls a
database, business-service hostname, ZaloPay endpoint, or AI provider directly.

4. User functionality
---------------------
Implemented User Service capabilities include:

- student-only public registration;
- real bcrypt login and HS256 JWT issuance;
- profile read/update and password change;
- administrator user list/filter, status and role management;
- login-audit activity reporting.

After successful login, User Service publishes UserLoggedInEvent on the durable RabbitMQ
topic exchange lms_events using routing key user.loggedin. The event excludes passwords,
password hashes, and JWTs. Broker publication failure is non-critical to a valid login.

5. Course and learning functionality
------------------------------------
Implemented Course Service capabilities include:

- published catalog with search, category, minimum-price, and maximum-price filters;
- published course detail and category list;
- instructor-owned draft creation/list/update/publish;
- instructor-owned lesson create/list/update/delete for drafts;
- enrolled-course list, protected lesson/course learning data, resource URLs;
- idempotent lesson completion and persisted course progress;
- internally authenticated course purchase metadata and enrollment activation;
- student exam-access and instructor ownership APIs used by Exam Service;
- instructor enrollment/progress report;
- admin course report and course-status moderation;
- lesson-context AI request orchestration.

Course access requires both a published course and active enrollment. Student identity comes
from JWT. Payment Service is the only caller allowed to supply a student ID to the internal
enrollment endpoint, and that call requires the internal service secret. Enrollment is unique
per student/course and activation is idempotent.

When access is newly activated, Course Service publishes CourseAccessActivatedEvent with
routing key course.access.activated. Events do not perform a second enrollment write; the
synchronous internal HTTP activation is authoritative.

6. Exam and quiz functionality
------------------------------
Exam & Quiz Service implements:

- instructor-owned quiz create/list/detail/update/delete/publish for a draft course;
- single-choice question authoring within the quiz transaction;
- published quiz listing/loading for an actively enrolled student;
- secure student payloads without correct answers;
- server-side validation, grading, result persistence, and one submission per student/quiz;
- own-result list/detail for students;
- instructor-owned course quiz-result summary.

Exam Service obtains course and enrollment authorization from Course Service over HTTP with
the original JWT. It never reads Course DB. Browser-supplied student IDs, scores, pass flags,
or answer keys are not authoritative.

7. Payment functionality
------------------------
The current executable provider integration is ZaloPay Sandbox v2:

- POST /payments/checkout creates a backend-priced pending Payment DB transaction and signs
  the sandbox create request.
- GET /payments/check-status/:appTransId verifies payment ownership and signs a sandbox query.
- POST /payments/callback/zalopay validates the Key2 MAC and stored transaction identity.
- GET /payments/mine and owned payment detail expose student payment history.
- mock completion is disabled and returns HTTP 410.

Course price comes from Course Service, not the browser. A provider-confirmed pending payment
transitions to the canonical success state and publishes PaymentSucceededEvent using routing
key payment.succeeded. Payment Service then synchronously calls Course Service to activate
access. A failed pending payment publishes PaymentFailedEvent using routing key payment.failed
and never activates enrollment. Repeated confirmation is safe and does not duplicate the
payment transition or enrollment.

8. Reporting functionality
--------------------------
There is no Reporting or Analytics deployable service.

- Payment Service owns the admin revenue report. It reads Payment DB and calls the protected
  Course Service title endpoint for minimal metadata. Only status=success contributes revenue.
- Course Service owns admin course/enrollment reporting and instructor learning progress.
- Exam Service owns instructor quiz-result reporting.
- User Service owns login activity reporting.

Each owner queries only its own database.

9. AI learning support
----------------------
The actual path is:

Lesson UI -> API Gateway -> Course Service -> Course DB context -> External AI Chatbot System.

Course Service validates the student JWT, published course, active enrollment, lesson, and
question before sending course/lesson/resource/progress context. The external adapter reads
AI_API_KEY from its own server environment and calls the configured real provider. It returns
AI_PROVIDER_NOT_CONFIGURED when no key is present and never fabricates a canned answer. The
provider key is not passed to Web Client, API Gateway, or Course Service.

10. Message Broker
------------------
RabbitMQ is the implemented broker. The durable topic exchange is lms_events. Current routing
keys are:

- user.loggedin
- payment.succeeded
- payment.failed
- course.access.activated

Each JSON event has eventId, eventType, eventVersion=1, occurredAt, source, and data. Messages
are persistent and published with confirms/reconnect behavior. Database state is authoritative;
publication errors are logged safely, and consumers are expected to be idempotent.

11. Local Docker deployment
---------------------------
docker-compose.yml deploys Nginx, Web Client, API Gateway, four core services, four separate
MySQL containers/volumes, RabbitMQ, the external AI adapter, and the local payment-provider
compatibility fixture. Service-to-service traffic uses Docker DNS names, never localhost.

The optional database-backup-server profile creates timestamped, non-destructive SQL dumps in
its backup volume. It is disabled during normal startup.

Use:

  docker compose up -d --build

or double-click start-lms.bat. Open http://localhost:8080. Stop without deleting data using:

  docker compose down

Never use docker compose down -v for normal operation.

CREDENTIAL-DEPENDENT LIVE VERIFICATION
======================================

Source integration is complete for ZaloPay Sandbox and the external AI provider, but live
verification requires locally configured non-placeholder environment variables. No real keys
or secret values belong in this repository or this document.

- ZaloPay live sandbox create/query requires ZALOPAY_APP_ID, ZALOPAY_KEY1, ZALOPAY_KEY2.
- Live AI answers require AI_API_KEY.

The local external payment compatibility container is not the main payment-success path. The
main Web Client flow uses ZaloPay Sandbox checkout/status polling.

NOT IMPLEMENTED NOW / POSSIBLE FUTURE OPTIONS
=============================================

- Mobile Client remains a source/deployment placeholder and is not containerized as a working
  Flutter application.
- Momo is an architectural payment alternative; no live Momo adapter is currently claimed.
- Kubernetes, AWS/RDS, S3, CloudFront, production TLS, multi-region deployment, and managed
  broker/database services are future deployment options, not current runtime facts.
- Password-recovery email, discussion forums, and announcement delivery are not active APIs in
  the current source. Adding them must stay inside an approved existing owner and must not add
  a new core service or database.
