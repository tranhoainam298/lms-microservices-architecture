This audit records the product baseline before the business CRUD and demo-data pass. A UI entry is not counted as implemented unless it calls an owning-service API and persists or reads the owning database. Final evidence is recorded in `BUSINESS_FUNCTIONALITY_VERIFICATION.md` after focused tests and Docker E2E complete.

## Baseline conclusions

- The secure core is already real: identity, ownership, payment amount, enrollment, lesson progress, grading, and report authorization are backend-authoritative.
- The largest baseline gap is demonstrability: current fresh seeds are small, some strings are mojibake, existing named volumes do not automatically receive init-file additions, and several dashboards lack aggregate response fields.
- The implementation pass must therefore add an explicit idempotent current-volume seed command, retain equivalent fresh-volume seed support, add only backward-compatible aggregate/query fields, and wire active pages to those real APIs.
- Standalone category CRUD is intentionally not claimed while category remains a course-owned string. The existing real admin category update is retained; a fake category page will not be added.
- Live ZaloPay and AI provider verification remain dependent on external credentials and will not be represented by seeded provider success or canned answers.
