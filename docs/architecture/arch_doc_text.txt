1. Problem Requirements
1.1. Problem Description
1.1.1. System Context
In the context of the ongoing digital transformation in education, the shift from traditional training models to an online learning platform (LMS) is not merely a change in tools, but a strategic move to optimize resources and digitize knowledge assets. Currently, traditional practices still face several critical bottlenecks: learning materials are stored in a fragmented manner, tracking learner progress is challenging, and the data reporting processes remain manual and delayed.
To thoroughly resolve these issues, the Learning Management System (LMS) is proposed to break down geographical barriers and personalize the learning path. The platform will comprehensively digitize the training process through modern modules, including online payments, automated assessments, an administrative reporting system, and most notably, the integration of a 24/7 AI Chatbot for learning support. Ultimately, this system will deliver a superior operational advantage for the entire educational ecosystem.
1.1.2. The System's Objectives
The primary objective of the project is to develop a modern online training and management platform (E-Learning/LMS) that centralizes, accelerates, and streamlines the learning, teaching, and administrative processes. Specifically, the system aims to achieve the following core objectives:
Centralized Online Learning Platform: Create a flexible learning environment where learners can easily register accounts, explore and enroll in courses, and track their personalized development paths anytime, anywhere.
Standardization and Optimization of Training Processes: Empower instructors to effortlessly manage course content and question banks, while providing administrators with robust tools for user management, role-based access control, and effective system moderation.
Quality Assessment and Tracking: Build a comprehensive examination system with an automated grading mechanism, delivering intuitive statistical reports on learning progress and academic outcomes.
Digitization of Financial Transactions: Integrate a secure online payment gateway to automate course registration and streamline the payment processing workflow.
Pioneering AI Integration: Incorporate an AI Chatbot that serves as a virtual assistant, providing 24/7 support to answer queries, guide learning journeys, and recommend tailored content to users.
Sustainable and Robust Architecture (NFRs): Design a highly reliable system that ensures scalability, stringent data security, and seamless maintainability for future expansions.
1.1.3. Scope Of The System
The system is limited to the following core business and technical areas:
1.1.3.1. Scope for Students
Account management (Registration, login, password recovery).
Search, browse, and filter course listings.
Register and pay for online courses.
View course details, access lectures (videos, documents), and track progress.
Take quizzes/essays and view your learning results.
Interact, discuss, and use AI chatbots to answer questions.
1.1.3.2. Scope for Teachers
Create, edit, update, and delete courses.
Manage the lecture resource library (upload videos, documents).
Build a question bank and set up exams/tests.
Monitor student progress and view test results.
Participate in discussions and directly answer student questions.
1.1.3.3. Scope for Admin
Comprehensive user profile management (account locking/unlocking, system permissions).
Manage and categorize course catalogs.
Review course content before publication.
Extract in-depth statistical reports on user traffic, course coverage, and revenue.
Set up and manage system notification flows.
1.1.3.4. Architectural and engineering scope
Web/Mobile Client: Modern user interface with integrated glassmorphism UI/UX effects, developed on the React platform.
API Gateway & Backend Services: Business modules handling core logic, robust and synchronized operation on Node.js and C# platforms.
Database: Database management system for storing system information.
Peripheral Integration System: Payment gateway (ZaloPay/Momo) for transaction processing and AI Chatbot system to support learning.
Storage Infrastructure: Dedicated multimedia resource storage system for documents and lecture videos.

The specification tables below follow the main use cases from the original report. The business flows are preserved and placed into the required Software Architecture assignment format.
1.2.2.1. Course Management Activity
Use Case Name
Course Management
Description
The instructor creates a new course, updates course information, and authors lecture content including videos, documents, and quiz questions.
Actors
Instructor / Teacher
Input Data
Course information including title, description, category, price, cover image; lecture content including videos and documents; optional quiz questions.
Output Data
The course is saved into Course DB with a status such as Draft, Pending Review, or Published.
Main Flow
1. The instructor logs in to the system.2. The instructor selects Create New Course.3. The instructor enters basic course information such as title, description, category, price, and image.4. The instructor adds lessons by uploading videos or attaching documents.5. The instructor may add quiz questions for lessons or the whole course.6. The instructor clicks Save.7. The system stores the course data in Course DB.8. The system displays a course creation success message.
Alternative Flow
1. Save Draft: if the content is incomplete, the instructor saves the course as Draft for later editing.2. Publish Now: if approval is not required, the instructor publishes the course immediately.3. Edit Course: the instructor selects an existing course, updates information or lessons, and saves again.
Exception Flow
1. Missing required fields: the system displays an error and asks the instructor to complete the information.2. Invalid video/document format or size: the system rejects the upload and displays an error.3. Duplicate course name: the system informs that the course name already exists.
1.2.2.2. Learning Management Activity
Use Case Name
Learning Management
Description
The student accesses lecture content such as videos and documents. The system records learning progress and supports questions through AI.
Actors
Student, AI Chatbot System as supporting actor
Input Data
Lesson request with lesson ID, completion confirmation, and student learning questions.
Output Data
Lesson content from Course DB, updated learning progress, and AI support response.
Main Flow
1. The student selects a lesson in an enrolled course.2. Course Service receives the lesson request through API Gateway and retrieves content from Course DB.3. Course DB returns the lecture content such as video/document information.4. The system displays the video/document for the student.5. After studying, the student confirms lesson completion.6. The system records and updates learning progress.7. The system displays an update success message.8. If the student has a question, the question is sent to the AI Chatbot System.9. The AI Chatbot processes the question and returns a support answer.
Alternative Flow
1. The student revisits a completed lesson: the system allows viewing without duplicating progress.2. Automatic completion: the system marks the lesson complete when the video reaches the end.3. AI cannot understand the question: the system returns a default message and suggests contacting the instructor.
Exception Flow
1. Student is not enrolled: the system denies access.2. Lesson content is unavailable: the system displays a try-again-later message.3. Connection is lost: the system saves temporary progress and asks whether to continue after reconnection.
1.2.2.3. Question Bank and Exam Management Activity
Use Case Name
Question Bank and Exam Management
Description
The instructor authors questions and answer keys. Students retrieve quizzes, submit answers, and the system automatically grades and stores results.
Actors
Instructor / Teacher for question authoring; Student for quiz taking
Input Data
From instructor: question content, options, correct answer, difficulty, topic. From student: quiz request and answers.
Output Data
Questions and answer keys are saved in Exam DB. Quiz scores and attempt results are saved and displayed to students.
Main Flow
1. The instructor opens Question Bank Management.
2. The instructor enters question content and question type.
3. The instructor enters options and correct answers if applicable.4. The instructor assigns questions to a topic, course, or lesson.5. Exam & Quiz Service saves the question to Exam DB.
6. The student opens a quiz in an enrolled course.
S 7. Exam & Quiz Service loads questions from Exam DB.8. The student submits answers.9. Exam & Quiz Service compares submitted answers with the answer key.10. Exam & Quiz Service calculates score and saves the quiz result.11. The system displays score and result to the student.
Alternative Flow
1. The instructor edits or deletes unused questions.2. The instructor imports questions in bulk.3. The student retakes the quiz if allowed.4. The system automatically submits the quiz when time runs out.
Exception Flow
1. Missing question or answer data: the system asks for required data.2. Duplicate question: the system informs that the question already exists.3. Invalid import file: the system displays an import error.4. Quiz has no questions: the system displays an unavailable quiz message.
1.2.2.4. Payment Management Activity
Use Case Name
Course Payment
Description
The student pays for a course. The system sends the transaction to the external payment gateway, verifies the result, records the payment, and activates course access after success.
Actors
Student, External Payment Gateway
Input Data
Payment method, card/e-wallet information, course ID, and amount.
Output Data
Payment transaction status saved in Payment DB, course access saved in Course DB, and payment result notification shown to the student.
Main Flow
1. The student selects a paid course and proceeds to payment.2. The student selects a payment method and confirms payment.3. Payment Service creates a pending payment in Payment DB.4. Payment Service sends the request to Payment Gateway ZaloPay/Momo.5. The gateway returns the transaction result.6. If successful, Payment Service updates the status as Paid.7. Payment Service requests Course Service to activate course access.8. Course Service saves enrollment/access in Course DB.9. The system shows a payment success message.
Alternative Flow
1. Payment failed: the system shows an error and allows trying another method.2. Free course: payment processing is skipped and access is activated directly.3. Student cancels payment: the use case ends without saving a completed transaction.
Exception Flow
1. Payment gateway connection fails: the system displays a gateway connection error.2. Invalid payment information: the system rejects the request.3. Transaction timeout: the system cancels the request and asks the student to try again.
1.2.2.5. Reporting and Statistics Activity
Use Case Name
Reporting and Statistics
Description
The system provides reports for different actors: students view personal learning outcomes, instructors monitor class progress, and admins view overall system reports.
Actors
Student, Instructor / Teacher, Admin
Input Data
Student report requests, instructor class progress requests, and admin management report requests.
Output Data
Student grade/history view, instructor teaching report, and admin revenue/user/course reports.
Main Flow
1. Student opens personal statistics to view grades and progress.2. Teacher opens class statistics to monitor completion rate and average scores.3. Admin opens system reports and selects a reporting period or course scope.4. Payment Service queries paid transactions from Payment DB.5. Course Service provides course information from Course DB.6. The system calculates statistics and displays report charts or tables.7. Admin may export the report if required.
Alternative Flow
1. Student filters results by course or term.2. Teacher exports class progress as Excel.3. Admin schedules periodic report generation.

1.3. System Constraints
1.3.1. Technology Constraints
Code
Constraint
Description
TC01
Microservices
The system must separate business domains into independent services: User, Course, Exam & Quiz, and Payment.
TC02
API Gateway
All requests from Web/Mobile Client must go through the API Gateway for routing, JWT authentication, and rate limiting.
TC03
Database per Service
Only the databases in the selected architecture are used: User DB, Course DB, Exam DB, and Payment DB. No extra database is created outside the architecture.
TC04
REST + Event-driven
REST is used for immediate responses; Message Broker is used for publish/subscribe events such as login, payment, and course events.
TC05
External Integrations
Payments go through Payment Gateway ZaloPay/Momo; learning questions go through the AI Chatbot System.
TC06
Deployment
Services should be packaged with Docker and deployed using Docker Compose or Cloud; database backup is required for each service database.
1.3.2. Business Constraints
Code
Constraint
Description
BC01
Role-based Access Control
Students, instructors, and admins can only access functions that match their roles.
BC02
Lesson Access
Students must log in and enroll in a course before viewing lesson content.
BC03
Course Creation
Instructors must provide required information before saving a draft or publishing a course.
BC04
Quiz
A quiz must have valid questions and answer keys before students can take it.
BC05
Payment
Course Service only activates course access when payment succeeds. Failed payments must not create learning access.
BC06
Reporting
Admin can view system-wide reports; instructors can view only data related to their courses when the function is available.
BC07
AI Support
AI Chatbot is a support tool only and does not replace official responses from instructors or admins.

2. Architecture Design
2.1. Architecture Style Selection
The system adopts Microservices Architecture combined with an API Gateway and event-driven communication. This architecture is suitable for the LMS because its business domains can be separated independently: User, Course, Exam & Quiz, Payment, and AI Support. Each service has a clear responsibility and owns its data, improving scalability, maintainability, and fault isolation.
Code
Architectural Decision
Explanation
AD01
Microservices Architecture
Separates business domains by service, suitable for an LMS with multiple independent functional groups.
AD02
API Gateway
Acts as the single entry point for Web/Mobile Client; handles routing, JWT authentication, and rate limiting.
AD03
Database per Service
User DB, Course DB, Exam DB, and Payment DB are managed by their corresponding services.
AD04
Message Broker
Supports publish/subscribe for events that do not require direct synchronous communication.
AD05
External Services
Payment Gateway ZaloPay/Momo and AI Chatbot System are placed outside the core system.

2.8. Advantages and Disadvantages of Microservices Architecture
Type
Analysis
Advantages
Independent scalability; good fault isolation; easier change per business domain; suitable for Payment Gateway, AI Chatbot, and Message Broker integration.
Disadvantages
More complex operation than monolithic architecture; requires API contract management, logging, monitoring, distributed data handling, and control of network latency between services.

3.3. Software Design Principles
Separation of Concerns: each service handles only one clear business domain.
Single Responsibility: User Service does not handle courses, and Payment Service does not handle quizzes.
DRY: shared API response formats, error formats, and configurations can be reused through internal utilities when needed.
API Contract: services communicate through clear API and event contracts.
Database Ownership: a service must not directly access a database owned by another service.
4. System Deployment
4.1. Practical Deployment Architecture
The system can be deployed in a simulated environment using Docker Compose or on the cloud. The API Gateway is the single entry point. Each service runs in its own container and connects to its corresponding database. Database backup is performed per database: User DB Backup, Course DB Backup, Exam DB Backup, and Payment DB Backup.

4.3. Deployment Process
1. Prepare source code and environment configuration files (.env) for each service.
2. Initialize databases: User DB, Course DB, Exam DB, and Payment DB.
3. Start the Message Broker.
4. Build Docker images for the API Gateway and microservices.
5. Run the whole system with Docker Compose or deploy it to the cloud.
6. Check each service health endpoint and test the main flows: login, save draft course, view lesson, take quiz, payment, report, and AI question.
